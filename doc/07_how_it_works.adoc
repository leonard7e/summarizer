== How It Works

This chapter explains the internal mechanics of `summarizer` — how it splits input files into batches, calculates the available token budget, and builds the prompt that is sent to the model.

=== High-Level Flow

[source]
----
  CLI args & config
        │
        ▼
  ┌─────────────────┐
  │  Parse model ID │  "google:gemini-1.5-flash"  →  provider="google", model="gemini-1.5-flash"
  └────────┬────────┘
           │
           ▼
  ┌─────────────────────────┐
  │  Query context window   │  GET /models  →  api_limit (tokens)
  └────────┬────────────────┘
           │
           ▼
  ┌──────────────────────────────────────────┐
  │  Batch files                             │
  │  • Estimate file size in chars           │
  │  • Group files until budget is exhausted │
  │  • Start a new batch when full           │
  └────────┬─────────────────────────────────┘
           │
           ▼
  ┌──────────────────────────────────────────┐     ┌────────────────────┐
  │  For each batch:                         │────▶│  LLM API call      │
  │    build_prompt(instruction,             │     │  provider.complete()│
  │                 files,                   │◀────│  → new_result      │
  │                 previous_result)         │     └────────────────────┘
  └────────┬─────────────────────────────────┘
           │  previous_result = new_result
           │  (next iteration)
           ▼
  ┌─────────────────┐
  │  Print result   │  stdout
  └─────────────────┘
----

=== Step 1: Model ID Parsing

The model string passed via `--model` (or from `default_model` in the config) is split on the first `:` into a _provider_ name and a _model ID_:

[source]
----
"openrouter:anthropic/claude-3-haiku"
 ^^^^^^^^^^  ^^^^^^^^^^^^^^^^^^^^^^^^
 provider    model (everything after the first ':')
----

An error is returned if no `:` is present.

=== Step 2: Context Window Query

Before batching, `summarizer` queries the provider's API for the context window of the chosen model (in tokens). This value is used as the upper bound (`api_limit`) for all budget calculations.

If the API does not report a context window, the fallback is:

[source,rust]
----
pub const DEFAULT_CONTEXT_LIMIT: usize = 8_192; // tokens
----

=== Step 3: Token Budget Calculation

Before every batch, the amount of file content that can safely be included is computed by `compute_file_budget`:

[source]
----
file_budget (chars) = (api_limit
                       - instruction_tokens
                       - previous_result_tokens
                       - max_output_tokens
                       - OVERHEAD_TOKENS
                      ) × CHARS_PER_TOKEN
----

Where the constants are:

[cols="1,1,3", options="header"]
|===
| Constant | Value | Meaning

| `CHARS_PER_TOKEN`
| `4`
| Rough estimate of UTF-8 characters per token (adequate for typical prose and code).

| `OVERHEAD_TOKENS`
| `512`
| Fixed reservation for JSON structure, role tags, separators, and other API overhead.

| `max_output_tokens`
| `2048` (default)
| Tokens reserved for the model's response. Configurable via `max_output_tokens` in `config.yaml`.
|===

[NOTE]
====
The budget is recalculated before each batch because `previous_result` grows over iterations.
A larger previous result leaves less room for file content in the next batch.
====

=== Step 4: Batching Algorithm

Files are grouped into batches using a single left-fold over the file list.
The initial file budget (computed with no previous result yet) determines the maximum batch size:

[source]
----
For each file:
  estimate size ≈ file size in bytes (≈ chars for UTF-8)
  if current_batch is non-empty AND current_size + file_size > initial_budget:
      start a new batch
  else:
      append file to current batch
----

This is a simple greedy _first-fit_ algorithm. Files are never split; a file that is larger than the budget on its own occupies a batch by itself.

=== Step 5: Prompt Construction

For each batch, `build_prompt` assembles the text that is sent to the model:

[source]
----
<instruction>

--- Bisheriges Ergebnis ---      ← only present from the 2nd batch onward
<previous_result>

--- Datei: <filename> (Encoding: utf-8) ---
<file content>

--- Datei: <filename> (Encoding: utf-8) ---
<file content>
...
----

The instruction comes from `--prompt`, `--prompt-file`, or the built-in default.
The previous result section is omitted for the very first batch.

=== Step 6: Iteration and Output

Each API call returns a new result string, which becomes the `previous_result` for the next batch.
After all batches have been processed, the final result is printed to standard output.

Progress percentages are printed to standard error so that the summary can be redirected cleanly:

[source,bash]
----
summarizer *.txt > summary.txt
# stderr: 0% 33% 66% 100%
# stdout: final summary (written to summary.txt)
----
